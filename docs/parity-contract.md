# Go Conversion Parity Contract (Slice 1)

This document locks the current Python static analyzer as the parity oracle for
the upcoming Go port. Future Go implementations must reproduce the committed
golden JSON for every synthetic fixture in `tests/fixtures/workbooks/` unless
this contract is intentionally revised.

## Oracle

- **Python analyzer** (`spreadsheet_auditor.audit.audit_workbook`) is authoritative
  for conversion-stage parity.
- Goldens are generated by `scripts/emit_goldens.py` and verified with
  `make verify-goldens`.
- Regression tests in `tests/test_golden_parity.py` compare normalized JSON bytes
  for every committed fixture.

## Normalized Report Fields

| Field | Parity rule |
| --- | --- |
| `workbook_path` | Always the literal string `"<workbook>"` in goldens and parity tests |
| `supported_format` | Lowercase suffix of the scanned file (`.xlsx` or `.xlsm`) |
| `summary` | Counts derived from the scanned workbook; rollup keys sorted alphabetically |
| `sheets` | Workbook sheet order from the parser |
| `issues` | Sorted by Python lexical `(sheet, cell, rule_id)` |

## Issue Ordering

Issues are **not** sorted by natural cell coordinates. Coordinates are compared as
strings, so `B10` sorts before `B2`. The `lexical_cell_ordering` fixture locks this
behavior for Go work.

## Sheet States

Canonical `sheets[].state` values emitted today:

- `visible`
- `hidden`
- `veryHidden`

The `hidden_and_very_hidden` fixture commits all three states.

## Used Range

`used_range` follows the current Python/openpyxl `Worksheet.calculate_dimension()`
output for each sheet. Go parity should match committed golden values rather than
re-deriving bounds independently.

## External Workbook References

External reference detection preserves the current Python regex behavior:

- Pattern: `\[[^\]]+\]` anywhere in the formula text
- `details.references` lists unique bracketed workbook tokens, sorted

Do not broaden matching for parity (for example, path-only links without brackets)
unless this contract and goldens are updated together.

## Formula Parse Errors

`FORMULA_PARSE_ERROR` may be parser-specific when Go adopts a different tokenizer.
For parity, compare `FORMULA_PARSE_ERROR` findings only on formulas where committed
goldens intentionally lock behavior. Other rules should still match on the same
cells.

## Formula Pattern Anomaly

The `formula_anomaly` and `consistent_copied_formulas` fixtures lock:

- positive detection for a single local outlier in a copied column run
- no anomaly noise for a consistent copied column

Normalization uses `spreadsheet_auditor.formula_pattern.normalize_formula` (lexical,
position-relative, not evaluated).

## Escaping And Workbook-Provided Text

The `escaping_workbook_text` fixture locks sheet names and formulas containing
`<script>` in JSON output. HTML review-pack escaping is covered in Python CLI tests
and Go `internal/reviewpack` tests; JSON goldens still contain raw workbook strings.
Go HTML parity uses structural/content matching (sections, escaped workbook text,
severity classes) rather than byte-identical HTML across languages.

## Go Export Contract (HTML And CSV)

Go exports are rendered from the in-memory `AuditReport` without re-reading JSON.

| Field / rule | Parity rule |
| --- | --- |
| `exported_at` | RFC3339 UTC in HTML metadata and the first CSV column |
| `issue_id` | `rule_id\|sheet\|cell\|message\|suffix` for export selection. The suffix is an 8-hex deterministic hash of formula/details evidence. |
| CSV header order | Fixed: `exported_at`, `workbook_path`, `supported_format`, `issue_id`, `priority`, `impact_factors`, `severity`, `category`, `rule_id`, `title`, `sheet`, `cell`, `formula`, `message`, `rationale`, `remediation`, `details_json` |
| `details_json` | Sorted object keys, `SetEscapeHTML(false)`, no trailing newline |
| CSV injection | Prefix values starting with `=`, `+`, `-`, `@`, tab, or carriage return with `'` before `encoding/csv` writes the field |
| Selected export | Backend filters by `issue_id`; unknown IDs are ignored |

`SaveReviewPack` remains a deprecated HTML wrapper that uses the current UTC time for
`exported_at`. Prefer `SaveExport` with format `html` or `csv`.

## Macro-Enabled Workbooks

The `static_xlsm` fixture locks static `.xlsm` parsing. Parity does **not** require
macro execution, VBA analysis, or link refresh.

## Empty And Minimal Workbooks

- `empty_workbook`: default sheet with no formulas
- `single_value_cell`: one non-formula value cell

These lock baseline inventory behavior for sparse workbooks.

## Sparse Sheet Iteration (Go)

The Go analyzer walks worksheet rows with Excelize `GetRows()`, which is built
on the library's `Rows()` / `Columns()` iterator, instead of iterating every
coordinate inside `GetSheetDimension()`. This keeps bloated declared dimensions
from triggering a full rectangular cell walk while preserving actual row indexes
for sparse populated rows.

Committed goldens for existing fixtures remain unchanged.

## Excel Error Sentinels (Go-only until Python parity)

Go adds these rules beyond the existing `#REF!` pair:

| Rule | Trigger |
| --- | --- |
| `EXCEL_ERROR_VALUE` | Displayed `#DIV/0!`, `#VALUE!`, `#NAME?`, `#N/A`, `#NUM!`, `#NULL!`, `#SPILL!`, `#CALC!` |
| `EXCEL_ERROR_FORMULA` | Stored formula text contains one of the same sentinels |

`#REF!` continues to use `BROKEN_REF_VALUE` / `BROKEN_REF_FORMULA` for golden
parity. New rules emit `details.error_code` with the exact sentinel string.

Python parity for the new rules is not part of the committed golden gate yet;
Go unit tests in `internal/audit/sheet_scan_test.go` cover sparse iteration and
error detection.

## Analyst Priority Bands

Go and Python both add derived analyst routing fields on each issue:

| Field | Meaning |
| --- | --- |
| `priority` | Deterministic band: `critical`, `high`, `medium`, or `low` |
| `impact_factors` | Sorted `{code, explanation}` entries explaining the band |

Rule `severity` is unchanged. Priority is assigned in `internal/priority` from
sheet visibility, output-like sheet names, external dependencies, formula-cluster
size, risky volatile functions, hardcoded constants, and isolated low-risk `TODAY()`
usage.

Committed analyzer JSON goldens under `tests/fixtures/golden/*.json` remain a
Python/Go parity contract. `make verify-goldens` compares generated workbook
fixtures and Python JSON to committed files, then runs the Go golden parity test
against the same committed JSON.

## Public / Customer Smoke Workbooks

Real customer or public sample workbooks are **out of scope** for committed goldens.
They may be used locally as informal smoke tests but are not part of the conversion
parity gate.

## Maintenance Commands

```bash
make regenerate-goldens   # rebuild fixtures and goldens
make verify-goldens       # fail if committed files drift
make check                # lint + full test suite including parity tests
```
